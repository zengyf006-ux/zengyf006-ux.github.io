# ATLAS X Pro 实时行情、K 线与专业交易体验系统级重构设计

日期：2026-07-11
状态：待用户书面确认后进入实施计划
目标分支：`atlas-x-pro-terminal`
线上目录：`atlas-x-pro/`

## 1. 背景与问题定义

现有 ATLAS X Pro 已具备较完整的模拟交易骨架，但“自动化测试通过”与“成熟交易终端体验”之间仍有明显差距。当前最严重的问题不是单个按钮或局部样式，而是行情数据、K 线状态、移动端交互和功能完整度之间缺少统一架构。

已经确认的核心缺陷：

1. 演示 K 线固定按 60 秒间隔生成，导致不同周期在降级场景下看起来相同。
2. 周期切换没有请求代际隔离，旧请求可能晚返回并覆盖新周期。
3. 首次 REST 请求、WebSocket 连接和失败降级的等待链过长，造成长时间加载。
4. QA 模式绕过真实网络，因此旧门禁不能证明真实行情在用户设备上有效。
5. K 线选中状态缺少完整退出路径，移动端容易永久黏住十字线。
6. 选中 K 线只展示少量信息，缺少完整 OHLC、涨跌额、振幅、成交量、成交额和指标值。
7. 图表缺少可视区最高价、最低价、当前周期倒计时、数据新鲜度等专业标记。
8. 主界面视觉层级、订单簿密度、下单路径和移动端空间利用仍明显低于成熟交易平台。
9. 行情、订单簿、逐笔成交和下单引用价格缺少统一的实时数据状态，可能出现区域间不一致。
10. 当前功能扩展以增量模块为主，缺少一次完整的产品级重新编排。

本轮不再采用“指出一个修一个”的补丁方式，而是按系统边界分阶段重构。

## 2. 产品目标

构建一套公开实时行情驱动、跨端一致、交互成熟的高保真模拟数字资产交易终端。

必须达到：

- 正常网络下，首屏迅速显示最近有效行情，不长时间空白转圈。
- 当前价格、订单簿、逐笔成交和当前 K 线来自同一数据会话。
- 每次切换交易对或周期都得到正确且可验证的新数据。
- 网络失败时明确显示过期或降级，不把随机演示价格伪装成实时价格。
- 手机端不是桌面页面缩小版，而是独立的交易流程。
- 功能扩展以真实交易任务为依据，不机械复制任何交易所品牌或完整布局。
- 继续保持纯模拟交易边界，不接真实资金、交易账户、API Secret、私钥或助记词。

## 3. 方案比较

### 方案 A：继续由浏览器直连多个交易所

优点：实现最轻，延迟最低，没有网关成本。

缺点：仍受地区网络、CORS、WebSocket 阻断、交易所域名可用性和浏览器限制影响；用户设备之间表现不一致；无法统一缓存、健康检查和数据格式。

结论：不采用为主架构，只保留为可选直连快路径。

### 方案 B：浏览器 + 轻量行情网关 + 直连快路径（推荐）

结构：

- GitHub Pages 继续托管前端。
- 在现有 Supabase 项目部署公共行情 Edge Function。
- 初始快照、历史 K 线、市场列表和健康状态统一走网关。
- 实时流优先尝试官方直连；直连失败时切换到网关 SSE 中继。
- 客户端统一消费标准化事件，不关心上游交易所字段差异。

优点：稳定性、性能和可维护性平衡最好；可以快速部署；不保存用户敏感数据；可以逐步增加上游数据源。

缺点：需要维护网关限流、缓存和健康探测；SSE 中继需要定期重连。

结论：本轮采用。

### 方案 C：完整独立后端交易平台

结构：持久化账户、服务端账本、撮合、实时网关、认证、审计和通知全部后端化。

优点：最接近正式商业交易平台。

缺点：范围巨大，涉及认证、数据库、幂等、风控、运维和安全；当前阶段容易把产品体验重构拖入长期后端工程。

结论：作为后续产品化阶段，不与本轮混做。

## 4. 总体架构

### 4.1 行情网关

部署目标：现有 Supabase 项目 `vtcunypvhtudragsittb`。

函数名称：`atlas-market-gateway`。

网关只处理公开行情，不接收或存储用户账户、订单、余额、密钥或个人信息。

接口：

- `GET /health`：上游可用性、延迟、最后成功时间和网关版本。
- `GET /markets`：标准化市场列表、24h 数据、价格精度、数量精度和状态。
- `GET /snapshot?symbol=BTCUSDT`：ticker、best bid/ask、简化深度、最近成交和数据时间戳。
- `GET /candles?symbol=BTCUSDT&interval=15m&limit=300`：标准化历史 K 线。
- `GET /stream?symbol=BTCUSDT&interval=15m`：SSE 实时流，事件类型为 `ticker`、`book`、`trade`、`kline`、`status`、`heartbeat`。

上游策略：

- 初期至少支持 Binance、OKX、Bybit 三个公开数据适配器。
- 每个请求按健康评分选择上游，不固定绑定单一交易所。
- 同一实时会话内，ticker、book、trade、kline 必须来自同一主上游，避免数据互相矛盾。
- 上游切换时发送明确 `status` 事件，并重建快照。
- 所有事件包含 `provider`、`serverTime`、`receivedAt`、`sequence` 和 `symbol`。

安全与资源控制：

- 公共函数关闭 Supabase JWT 校验，但实现严格的 Origin 白名单、方法限制、参数白名单、响应头和每 IP 轻量限流。
- 不允许任意 URL 代理，防止 SSRF。
- 最大 K 线数量、最大深度档位和 SSE 连接时长必须有限制。
- 任何上游错误都返回标准化错误，不回传内部堆栈或服务凭据。

### 4.2 客户端行情状态层

新增独立 `market-data-engine`，不继续把网络、图表、订单簿和 UI 全部塞在 `app.js`。

核心状态：

- `sessionId`：交易对和周期共同决定的实时会话。
- `requestGeneration`：每次切换交易对或周期递增，旧请求即使返回也必须丢弃。
- `connectionState`：`booting`、`live`、`reconnecting`、`stale`、`offline`。
- `provider`、`lastServerTime`、`lastReceivedAt`、`latencyMs`、`staleForMs`。
- `ticker`、`book`、`trades`、`candles` 使用同一会话版本。

缓存：

- 使用 IndexedDB 保存最近有效市场列表、每个交易对/周期的 K 线和最后快照。
- 页面启动时先渲染缓存，再后台刷新。
- 缓存必须显示数据年龄，不得冒充实时。
- 缓存按版本和时间淘汰，避免无限增长。

连接顺序：

1. 立即显示页面壳和本地缓存。
2. 并行请求网关快照和历史 K 线。
3. 建立实时流。
4. 实时流成功后原子替换为同一会话数据。
5. 直连官方 WebSocket可作为快路径；失败后自动使用网关 SSE。
6. 数据超过阈值未更新时标记 `stale`，不继续显示“实时”。

## 5. K 线模块重构

### 5.1 正确周期体系

首批支持：

- 1m、3m、5m、15m、30m
- 1h、2h、4h、6h、12h
- 1d、1w

每个周期必须有明确毫秒映射。任何降级或测试数据也必须按照真实周期生成时间戳，禁止固定 60 秒。

周期切换流程：

- 立即更新按钮选中态和加载状态。
- 取消旧 REST 请求、关闭旧实时流、递增 generation。
- 保留旧图作为弱化背景，不清空成白屏。
- 新周期首批数据到达后一次性切换。
- 旧 generation 的任何回包不得写入状态。
- 切换完成后验证首尾时间跨度与周期匹配。

### 5.2 选中 K 线信息

选中一根 K 线时展示：

- 完整时间和周期。
- 开盘、最高、最低、收盘。
- 涨跌额、涨跌幅、振幅。
- 成交量、成交额。
- EMA10、EMA20；若启用其他指标，展示对应值。
- 是否已收盘。
- 数据来源和时间戳。

移动端采用靠边信息卡，避免遮挡主要蜡烛；桌面端采用顶部数据条和轻量浮层组合。

### 5.3 选中与取消状态机

- 第一次点击或长按：选中最近蜡烛。
- 再次点击同一根蜡烛：取消。
- 点击绘图区空白：取消。
- 点击明确关闭按钮：取消。
- 按 Esc：取消。
- 切周期、切交易对、重置图表：取消。
- 手势拖动开始后：取消或进入平移状态，不能同时保留错误十字线。

### 5.4 专业图表标记

必须新增：

- 可视区最高价标记及引导线。
- 可视区最低价标记及引导线。
- 最新价格线、价格标签和当前 K 线倒计时。
- 数据过期标识。
- 可视区时间轴和价格轴自适应。
- “回到最新”按钮。
- 缩放、拖动、重置和全屏。

### 5.5 指标与工具

首批指标：

- MA、EMA、BOLL、VOL、MACD、RSI、KDJ。

规则：

- 主图指标和副图指标分组。
- 再次点击当前指标可关闭。
- 指标参数有默认值并可恢复默认。
- 手机端使用底部工具面板，不把桌面按钮硬压缩。

## 6. 移动端专业交易体验重排

### 6.1 信息层级

手机交易主屏按任务顺序组织：

1. 交易对、市场类型、实时价格和连接状态。
2. 周期、指标、图表工具。
3. K 线主视图。
4. 订单簿 / 逐笔成交切换。
5. 下单面板。
6. 持仓、委托、成交、审计和资产。

不允许同时在一屏堆满所有模块。使用标签、底部抽屉和全屏子视图按需展开。

### 6.2 订单簿

- 价格、数量、累计数量使用稳定列宽和等宽数字。
- 支持价格精度聚合。
- 支持单边买盘、单边卖盘和双边。
- 显示中间价、价差和深度比例。
- 支持点击档位回填订单价格。
- 移动端支持在不遮挡下单输入的情况下快速切换订单簿。

### 6.3 下单面板

现货模拟模式保留并完善：

- 市价、限价、止损限价、止损市价。
- 数量、金额和仓位比例切换。
- 可用余额、预估手续费、预估成交均价、预估滑点。
- 止盈止损和高级退出入口。
- 明确的确认、风险提示和审计入口。

第二阶段新增纯模拟永续合约模式：

- 全仓 / 逐仓。
- 杠杆选择。
- 开多 / 开空、平多 / 平空。
- 标记价格、指数价格、资金费率和倒计时。
- 预估强平价、保证金和手续费。
- 只做模拟，不接真实账户或资金。

## 7. 功能扩展路线

本次总重构分为四个可独立验收的子阶段，避免一次提交失控。

### 阶段 1：实时行情与 K 线正确性

- 行情网关。
- 客户端行情引擎。
- 缓存优先首屏。
- 周期切换正确性。
- K 线完整信息、取消交互、高低点和倒计时。

这是阻塞阶段，未通过不得进入后续功能堆叠。

### 阶段 2：移动交易主屏和订单簿重排

- 手机端视觉层级重构。
- 订单簿、逐笔成交和下单联动。
- 全屏图表和底部工具面板。
- 统一数字排版、间距和触控标准。

### 阶段 3：功能完整度扩展

- 专业市场筛选器。
- 更多指标和绘图工具。
- 下单类型完善。
- 模拟永续合约工作区。
- 持仓管理、保证金、强平预估和资金费率模拟。

### 阶段 4：性能、可靠性和产品化收尾

- 加载性能预算。
- 断网、弱网、上游切换和恢复测试。

- 视觉一致性、空状态和错误状态。
- 完整四端人工截图审查。

## 8. 性能目标

- 有有效缓存时，页面壳和旧快照目标 300ms 内可见。
- 正常网络下，首批新快照目标 1.5 秒内出现。
- 周期切换点击后 100ms 内出现明确反馈。
- 正常情况下，实时 ticker 延迟目标低于 1 秒。
- 超过 3 秒无更新显示重连；超过 10 秒显示数据过期。
- 切换周期时不允许整块图表长时间白屏。
- 图表拖动和缩放目标保持 50–60fps，低端手机不得持续掉到不可用水平。

这些是目标，不伪造测量结果；验收必须记录真实数据。

## 9. 验收体系重做

### 9.1 数据正确性

- 1m、30m、1d 的时间跨度和蜡烛时间戳必须不同且正确。
- 连续快速切换周期，最终只能显示最后一次选择。
- 旧请求晚返回不能覆盖当前周期。
- ticker、order book、trades、kline 属于同一 session 和 provider。
- 实时价格必须带服务器时间和接收时间。
- 数据过期后 UI 必须退出“实时”状态。

### 9.2 K 线交互

- 选中、同根取消、空白取消、关闭按钮、Esc、切周期取消全部测试。
- 信息卡字段完整且计算正确。
- 最高价和最低价标记与可视区数据一致。
- 最新价线和倒计时正确。
- 指标开启、关闭和参数恢复正确。

### 9.3 实网门禁

旧 QA 模式不得完全绕过网络验证。

新增两类门禁：

- 确定性模拟提供商测试：用于稳定 CI。
- 独立真实网络 smoke：验证网关健康、K 线周期差异、实时价格更新时间和故障状态；允许明确标记外部上游故障，但不能把假数据算作通过。

### 9.4 视觉与跨端

继续覆盖：

- iPhone 390 × 844
- iPhone 430 × 932
- Desktop 1440 × 900
- Desktop 1920 × 1080

每阶段必须：

1. 自动交互和布局测试。
2. 关键状态截图。
3. 人工审查信息层级、遮挡、触控、字体、密度和专业感。
4. 更新 `atlas-x-pro/PROJECT-STATE.md`。
5. 未全部通过不得部署。

## 10. 错误处理

- 首屏请求失败：显示缓存和明确失败状态，不空白等待。
- 实时流断开：保留最后数据、标记重连并指数退避。
- 上游切换：清理旧 sequence，重新请求快照，再接收新流。
- 数据字段异常：丢弃单条异常，不污染整个会话。
- 网关完全不可用：允许官方直连快路径；仍失败则进入离线状态，停止伪装实时更新。
- 缓存损坏：删除对应缓存分区，不清空模拟账户账本。

## 11. 不做事项

本轮不做：

- 真实资金充值、提现或真实下单。
- 真实交易所账户连接。
- API Key、Secret、助记词或私钥保存。
- 服务端真实撮合和清算。
- 复制 OKX、Binance、Bybit 等平台品牌、商标、完整视觉或文案。
- 为了功能数量在主屏无节制堆按钮。

## 12. 发布与回滚

- 所有开发继续在 `atlas-x-pro-terminal`。
- 行情网关先部署到独立函数版本，再由开发分支指向测试地址。
- 前端、网关和数据协议均带版本号。
- 每个阶段保存线上回滚点。
- 只有阶段门禁全部通过，才允许非强制快进 `main`。
- 若实时价格错误、周期错误、白屏或核心下单流程回归，立即回滚前端，并保留网关独立关闭能力。

## 14. 设计自检

- 没有 TBD 或未定义核心字段。
- 数据源、缓存、会话、失败状态和发布边界一致。
- 总体范围已拆分为四个可独立验收阶段。
- 第一阶段只解决实时行情和 K 线正确性，不同时引入合约账本，避免范围失控。
- 网关只处理公开行情，不触碰用户模拟账户和资金数据。
