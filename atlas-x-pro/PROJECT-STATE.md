## 2. 最终产品目标

开发一套达到成熟国际数字资产交易终端水准的高保真模拟交易产品。

ATLAS X Pro 不照抄任何一家交易所，而是按功能模块持续研究和对标全球成熟交易平台：吸收经过验证的业务逻辑、交互效率、信息层级、风险反馈、跨端设计和视觉优点；过滤广告化、营销入口、视觉噪声、功能堆叠、晦涩术语、移动端拥挤和高风险诱导。

衡量标准不是“功能数量最多”，而是：

1. 专业用户可以高效率看盘、制定计划、模拟下单和复盘。
2. 新用户不会因层级混乱、术语晦涩或操作反馈不足而误操作。
3. 桌面端具备成熟专业工作区的信息密度。
4. 手机端采用独立交易流程，不把桌面多栏粗暴压缩到小屏。
5. 每个新增功能必须解决明确交易任务，并通过自动测试和截图验收。
6. 任何功能都不能以牺牲稳定性、可理解性和交易安全为代价。

## 3. 长期对标方法

核心长期对标对象包括但不限于：

- Binance：订单区信息密度、价格/数量/总额联动、高级委托入口。
- OKX：交易前风险参数、止盈止损语义、执行上下文和跨端流程。
- Bybit：仓位计算、止盈止损联动、结果预览和持仓管理。
- Coinbase Advanced：清晰克制的表单分组、费用透明、可理解性和可访问性。
- Kraken Pro：专业桌面布局、紧凑数据表达、订单确认和审计逻辑。
- 其他成熟平台：仅在某个具体模块存在明显更优方案时加入对标，不因“平台多”而盲目复制。

对标纪律：

- 不复制品牌标识、品牌配色、专有文案或完整页面结构。
- 不把所有交易所功能机械叠加到一个页面。
- 每项功能先回答“解决什么交易任务”，再决定是否进入产品。
- 同类方案只保留最清晰、最高效、最稳的一种主流程；高级选项按需展开。
- 所有关键交互必须同时考虑桌面和手机。
- 每轮新增能力后重新检查信息密度、交互冲突和状态污染。

## 4. 不可突破的产品边界

- 当前为公开行情驱动的高保真模拟交易平台。
- 不接入真实资金。
- 不提供真实充值或提现。
- 不保存或要求用户输入交易所 API Key、Secret、助记词或私钥。
- 不连接真实交易所账户，不伪装成真实交易所。
- 所有模拟账户、订单、持仓、成交、账本和设置优先使用本地状态。
- 必须明确标注模拟环境和风险边界，不构成投资建议。

## 5. 不可回退的工程与设计约束

- 不使用 iframe 套旧版。
- 不引用 V7—V14 或旧 `atlas-x/` 的 DOM、CSS 和业务脚本作为新版运行依赖。
- 不出现“邓总”“项目展示”“CLIENT SHOWCASE”“Release Candidate”等演示稿文案。
- 不使用启动进度、粒子背景、无意义动效、玻璃卡片堆叠或大面积装饰渐变。
- 不为了显得功能多而加入尚未服务核心交易闭环的模块。
- 不直接在 `main` 或线上旧站试错。
- 未通过四种视口自动验收和人工截图审查，不得合并 PR。
- 样式和脚本必须按职责拆分，但禁止重新走向无限追加覆盖文件的补丁式架构。
- 不允许通过删除断言、跳过测试或放宽关键规则伪造绿灯。

## 10. 当前唯一优先任务

进入下一阶段：**订单与成交审计中心**。

目标不是再堆一个统计页面，而是让用户能回答：

1. 订单从创建、确认、挂单、部分成交到完成或取消，经历了什么。
2. 实际成交价格与下单参考价格之间产生了多少滑点，原因是什么。
3. 手续费、模拟资金费、滑点成本分别是多少。
4. 一个高级策略触发后，关联的子订单和持仓变化如何追溯。
5. 桌面与手机都能从订单、成交或持仓进入同一条审计记录。

本阶段门禁：

- 先完成模块设计与数据结构，不直接往页面堆 UI。
- 审计记录必须来自现有账本和订单状态，不维护第二套互相矛盾的数据源。
- 不影响现有撮合、余额、持仓、预留协调和退出策略。
- 新增自动测试覆盖桌面与手机，并重新跑完整四视口验收。
- 完成后再次更新本文件。

## 11. 后续路线

优先顺序：

1. 订单与成交审计：执行时间线、滑点归因、费用拆分。
